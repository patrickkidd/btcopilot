# SARF Improvement for gemini-3-flash-preview

**Date**: 2026-03-04
**Goal**: Improve SARF variable coding F1 that regressed after gemini-2.5-flash → gemini-3-flash-preview upgrade
**GT hash**: 2d2af9fb
**Model**: gemini-3-flash-preview (thinking=1024)
**Outcome**: Marginal improvement, converged after 6 iterations. SARF regression is primarily a model-level behavior difference, not prompt-addressable.

## Problem

After upgrading from gemini-2.5-flash to gemini-3-flash-preview:
- Aggregate F1 improved +6.7%, Events F1 improved +9.1%
- SARF regressed -35.9%: S -38.4%, R -29.3%, F -68.4%, A +15.0%

## Diagnosis

gemini-3-flash systematically codes S=up/A=up/F=down on events where GT has R=distance/overfunctioning/projection/inside. The model interprets interpersonal behavior as emotional states rather than specific clinical relationship patterns. This is a fundamental conceptual difference from gemini-2.5-flash.

## Key Findings

1. **gemini-3-flash SARF regression is model-level, not prompt-addressable.** The model fundamentally interprets withdrawing/overfunctioning as emotional states (anxiety, functioning) rather than relationship patterns. No prompt change tested could reliably fix this.

2. **Confirmed failed techniques for this model:**
   - Negative/restrictive examples → event suppression
   - Schema description changes (both restrictive AND additive) → under-extraction
   - WRONG/RIGHT example pairs → triggers SARF variable suppression
   - Expanded clinical definitions → don't improve target metric
   - Positive examples showing correct coding → hurt unrelated metrics

3. **Only additive, non-example guidance works.** The SARF Decision Guide (a prioritized checklist with no code examples) was the only change that didn't cause regressions. Even this was marginal.

4. **Stochastic variance dominates.** SARF F1 varies 10-15% across identical runs, making it difficult to distinguish real improvement from noise. Single-run comparisons are unreliable.

## Remaining Change

The SARF Decision Guide was added to `fdserver/prompts/private_prompts.py` after the "Common confusions" section in SARF VARIABLE CODING:

```
**SARF DECISION GUIDE** — for each shift event, ask in this order:
  1. Does it describe how person X BEHAVES TOWARD other people?
     → Code **relationship** (distance, overfunctioning, projection, etc.)
  2. Does it describe a PHYSICAL/MENTAL health change?
     → Code **symptom**
  3. Does it describe someone being WORRIED or NERVOUS?
     → Code **anxiety**
  4. Does it describe someone LOSING/GAINING personal direction?
     → Code **functioning**
  Most events in family discussions answer question 1 — relationship is the
  most frequently coded variable.
```

No changes to `btcopilot/btcopilot/llmutil.py`.

## Recommendation

The SARF regression is acceptable collateral for the Aggregate/Events improvements from gemini-3-flash. Further SARF improvement likely requires:
- Fine-tuning or few-shot learning (not prompt engineering)
- A different model for SARF coding (e.g., keep gemini-2.5-flash for Pass 2)
- Structural changes like a dedicated SARF-only pass after event extraction
