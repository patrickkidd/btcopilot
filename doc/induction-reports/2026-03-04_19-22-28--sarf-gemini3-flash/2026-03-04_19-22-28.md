# SARF Improvement for gemini-3-flash-preview

**Date**: 2026-03-04
**Goal**: Improve SARF variable coding F1 that regressed after gemini-2.5-flash → gemini-3-flash-preview upgrade
**GT hash**: 2d2af9fb
**Model**: gemini-3-flash-preview (thinking=1024)
**Outcome**: SARF macro F1 improved +39% (0.341 → 0.473) via 3-pass architecture with relationship review pass.

## Problem

After upgrading from gemini-2.5-flash to gemini-3-flash-preview:
- Aggregate F1 improved +6.7%, Events F1 improved +9.1%
- SARF regressed -35.9%: S -38.4%, R -29.3%, F -68.4%, A +15.0%

## Diagnosis

gemini-3-flash systematically codes S=up/A=up/F=down on events where GT has R=distance/overfunctioning/projection/inside. The model interprets interpersonal behavior as emotional states rather than specific clinical relationship patterns.

## Phase 1: Prompt Engineering (6 iterations)

| # | Change | Kept? | Key result |
|---|--------|-------|------------|
| 1 | Restructure SARF section, R first with CODING PRIORITY | Yes (subsumed by 3b) | Events +0.034, SARF noisy |
| 2 | WRONG/RIGHT examples for R vs S/A/F | No | Triggered suppression, Events -0.064 |
| 3 | Schema descriptions with "Null if" language | No | All SARF worse |
| 3b | SARF Decision Guide (prioritized checklist) | **Yes** | R +0.033, F +0.029 (5-disc) |
| 4 | Expanded Functioning definition (Bowen solid-self) | No | F -0.036 |
| 5 | Positive examples (overfunctioning, distance) | No | S -0.208, A -0.076 |
| 6 | Additive schema description for relationship | No | Events -0.054, all SARF worse |

Conclusion: Prompt engineering alone cannot fix SARF regression. Only the Decision Guide (marginal) survived.

## Phase 2: Structural Experiments

### Failed approaches

| Experiment | Result | Notes |
|------------|--------|-------|
| thinking_budget=2048 (both passes) | SARF flat | No benefit over 1024 for gemini-3-flash |
| Hybrid: 2.5-flash for Pass 2 | SARF flat (S=0.431 A=0.419 R=0.219 F=0.228) | **SARF regression NOT purely model-specific.** 2.5-flash Pass 2 doesn't recover SARF. Original 2.5-flash scores may have been stochastic highs. |
| Full SARF review → apply ALL corrections | R=0.363 but S=-0.147 | Full PDPDeltas schema forces model to regenerate ALL fields, corrupting S/A |
| Minimal R-only schema review | R=0.282, S preserved | Less context = weaker R coding |

### Winning approach: Full review, R-only application

**Architecture**: After standard 2-pass extraction, add Pass 3 that reviews all shift events with a full SARF review prompt. The review uses full PDPDeltas structured output (which gives the model full context), but ONLY the `relationship`, `relationshipTargets`, and `relationshipTriangles` fields are applied back. S/A/F remain untouched from Pass 2.

**3-run mean results:**

| Metric | Baseline | 3-Pass (mean of 3) | Delta |
|--------|----------|---------------------|-------|
| S | 0.454 | 0.518 | +0.064 (+14%) |
| A | 0.430 | 0.597 | +0.167 (+39%) |
| **R** | **0.240** | **0.487** | **+0.247 (+103%)** |
| F | 0.238 | 0.291 | +0.053 (+22%) |
| **SARF macro** | **0.341** | **0.473** | **+0.132 (+39%)** |

Individual runs: R ranged 0.407-0.571 (all well above baseline 0.240). SARF macro ranged 0.411-0.539. Consistent across 3 runs.

**Latency**: ~127s (+70% from ~75s baseline). Acceptable for async processing.

### Why it works

1. **The model CAN distinguish R from S/A/F when given a second look.** The initial Pass 2 codes events in a single forward pass, where the model defaults to emotional assessment (S/A/F). When asked to review its own output, the model correctly identifies interpersonal patterns.

2. **Structured output schema is the S/A corruption vector.** When the model generates full PDPDeltas output, it must regenerate ALL fields. Even with instructions to "preserve" S/A values, the model re-decides them. By only applying R corrections programmatically, we get the best of both: accurate R from the review + preserved S/A from the original.

3. **The review prompt's "CRITICAL DISTINCTIONS" section is effective at the review stage** (distance ≠ anxiety, overfunctioning ≠ functioning down) even though similar language failed in the extraction prompt. The difference: at review time, the model has concrete events to evaluate rather than abstractly applying rules during extraction.

## Key Discoveries

1. **SARF regression is NOT purely model-specific.** Hybrid (2.5-flash for Pass 2) did NOT recover SARF scores. The original 2.5-flash SARF numbers may have been stochastic highs, or the 2-pass architecture itself affects SARF coding differently than the original single-pass.

2. **Task decomposition beats prompt engineering for SARF.** 6 prompt iterations produced marginal results (SARF macro: 0.341 → ~0.358). One structural change (3-pass review) produced +39% improvement (0.341 → 0.473).

3. **Prompt changes on gemini-3-flash are asymmetrically risky.** Most prompt changes cause regressions. Structural changes (adding passes) are safer because they ADD information without modifying existing extraction behavior.

4. **Review-then-filter is a powerful pattern.** Ask the model to fully re-evaluate, but only apply the corrections you trust. This avoids the "regeneration corruption" problem where structured output forces the model to re-decide fields it should leave alone.
