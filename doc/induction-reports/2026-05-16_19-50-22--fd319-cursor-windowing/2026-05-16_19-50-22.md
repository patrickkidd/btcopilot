# FD-319 — Re-extraction cursor windowing (prompt change)

Date: 2026-05-16. Branch: FD-319. Prompt change: CURSOR_EXTRACTION_RULE_TEMPLATE
in `btcopilot/personal/prompts.py` (default; overridable via FDSERVER_PROMPTS_PATH),
appended to Pass 1 only when an accepted re-extraction cursor is active.

## Change

When a discussion has an accepted cursor, the full conversation is sent for
context but a per-call nonced marker delimits the already-committed prefix and
a cursor rule instructs Pass 1 to emit items only for content after the marker.
Mechanism is in `pdp._windowed_conversation` / `_two_pass_extract`; the rule
text is the prompt change governed by this report.

## Measurement (scenario: re-extraction with prior accepted content)

Baseline vs cursor, 6 GT discussions, 2 runs each, real Gemini, repair on,
prior half accepted then full conversation re-extracted:

| metric | baseline | cursor |
|---|---|---|
| already-committed events re-emitted (per disc) | 1.83 | 1.25 |
| already-committed people re-emitted | 0 | 0 |
| parent/child recall (re-extraction) | 0.63 | 0.73 |
| people F1 | 0.897 | 0.909 |
| pair-bonds F1 | 0.65 | 0.71 |

Cursor cuts re-emitted committed events ~⅓ and raises re-extraction parent/child
recall ~0.10, no F1 regression. It does NOT eliminate re-emission — the
deterministic committed-duplicate guard remains the safety net (this prompt
reduces how often it fires, it is not the safety mechanism).

## Standard-F1 timeseries impact

None. The cursor is inactive on fresh single-shot extraction (no accepted
cursor), so the 6-GT full-extraction F1 that feeds `f1_timeseries.json` is
unchanged. No timeseries entry is appended — it would not be a comparable
number. The re-extraction scenario above is tracked here, not in the timeseries.

## Negative result recorded alongside

A dedicated structural-completion pass was tried and rejected the same day
(no recall signal above noise, regressed aggregate/events) — see strategy doc
"things that failed" #18. The cursor, not a separate pass, is the lever for the
re-extraction structural-recall gap.

## Robustness

Marker carries a random per-call nonce; user/transcript text cannot forge the
boundary (test `test_marker_nonce_not_forgeable_from_user_text`). Windowing,
accept-driven advance, partial-accept, idempotent recommit covered by
`test_reextract_cursor.py` (11 tests). Adversarial review surfaced a
concurrency class of defects (concurrent extract / diagram-blob clobber) —
deferred to a separate FD-264 child, NOT addressed here.
