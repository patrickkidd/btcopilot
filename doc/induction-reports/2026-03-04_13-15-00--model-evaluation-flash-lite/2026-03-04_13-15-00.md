# Model Evaluation: gemini-3.1-flash-lite-preview + Thinking Budget

**Date**: 2026-03-04
**Trigger**: T7-20 (issue #59) — blocked by HTTP 500 errors from flash-lite API. Re-evaluate after T7-18 split extraction landed.
**Model**: gemini-3.1-flash-lite-preview (API name)
**Architecture**: 2-pass split extraction (Pass 1: people+bonds+structural events, Pass 2: shift events+SARF)
**Harness**: `run_extract_full_f1.py` — 6 GT discussions, description-free event matching
**Experiment runner**: `/tmp/run_experiments.py` (custom per-pass model/config override)

---

## Context

Issue #59 established a flash-lite baseline before the split extraction existed (single-prompt). That baseline showed:
- Events F1: 0.305, Aggregate: 0.630
- 500 errors on discussion 37 (Marcus) in 2/4 runs
- Verdict: "Not a viable replacement without prompt tuning for the lite model's weaker instruction following"

T7-18 (issue #94) split extraction into 2 passes. Hypothesis: smaller per-pass context might avoid the 500s and the split's focused prompts might play to flash-lite's strengths.

## Key Findings

### 1. thinking_budget=0 was the bottleneck, not the model

The single most impactful finding. Both models improve dramatically with thinking=1024:
- 2.5-flash: Events F1 0.265 → 0.378 (+43%)
- flash-lite: Events F1 0.154 → 0.368 (+139%)

**Mechanism**: Without thinking, flash-lite fixates on the first two entities in "extract PEOPLE, PAIR BONDS, and STRUCTURAL EVENTS" and skips structural events entirely. With thinking=1024, the model reasons about what qualifies as a structural event, checks dates, and produces them.

### 2. CORRECTION: thinking+structured_output is NOT catastrophic

The strategy doc (item #8 in "Things that failed") records from 2026-02-14:
> thinking_budget=1024 on gemini-2.5-flash — CATASTROPHIC latency: 29+ min per case, API hangs. Thinking+structured_output incompatible.

**This is no longer true.** All 14 experiment runs used thinking=1024 with structured JSON output (`response_mime_type="application/json"` + `response_schema`). Zero hangs. Average latency ~8s per pass (16s per discussion).

Likely causes of the Feb 2026 failure:
- Single-prompt architecture had much larger output schema + context
- Google may have fixed thinking+structured_output compatibility since Feb
- The 2-pass split reduces per-call output complexity enough to avoid the hang

### 3. Flash-lite with think=1024 matches 2.5-flash with think=1024

Events F1: 0.368 vs 0.378 — within noise across 3 vs 2 runs. People and Bonds essentially identical. The models converge when given a thinking budget.

### 4. Thinking budget sweet spot is 1024

Clear bell curve for flash-lite:
- 0 → 0.154
- 512 → 0.295
- **1024 → 0.368** (peak)
- 2048 → 0.298
- 4096 → 0.355

Below 1024: insufficient reasoning for event extraction. Above 1024: over-thinking degrades output (model second-guesses itself, drops valid events).

### 5. Hybrid per-pass model selection doesn't win

Flash-lite+think on both passes beats or matches all hybrid configs. The bottleneck was thinking budget, not model capability. Once thinking is enabled, flash-lite handles both passes adequately.

### 6. Temperature is irrelevant

0.0 vs 0.1 is noise. Not a lever worth tuning.

### 7. 500 errors fully resolved by split architecture

All 84 API calls (14 runs × 6 discussions × ~1 call per pass, with retries) completed successfully. The split extraction fixes the flash-lite API reliability issue by reducing per-call complexity.

### 8. Variance is meaningful — single runs are unreliable

lite think=1024 ranged from 0.336 to 0.418 across 3 runs (±12%). Always use multi-run averages. The strategy doc's recommendation of "confidence-based testing (3-5 runs)" is validated here.

## Comparison to Single-Pass (Hypothesis)

The old single-pass architecture (pre-T7-18):
- 2.5-flash: Events F1 ≈ 0.470 (from strategy doc), Agg ≈ 0.595
- flash-lite: Events F1 ≈ 0.305, Agg ≈ 0.630, with 500 errors

The split architecture with thinking=0 *hurt* Events F1 for both models vs single-pass. This is expected: separating events into two passes removes contextual anchoring — Pass 1 can't see SARF variables, Pass 2 can't see structural events.

**Thinking recovers most of the loss**: 2.5-flash goes from 0.265 → 0.378 (split+think), approaching the old single-pass 0.470. Flash-lite goes from 0.154 → 0.368 (split+think), *exceeding* its old single-pass 0.305.

**Would single-pass + thinking be even better?** Prediction: yes, it would combine contextual advantage with reasoning advantage. Estimated Events F1 ~0.45-0.50 for 2.5-flash, ~0.40-0.45 for flash-lite. However, this would re-introduce the 500 error risk for flash-lite on complex discussions. The split+thinking approach is the better architecture: competitive quality with zero API failures.

---

## Recommendations

1. **Immediate (one-line change)**: Set `thinking_budget=1024` in `gemini_structured()` in `llmutil.py:196`. This lifts Events F1 by +43% for the current 2.5-flash production config.

2. **Short-term**: Switch `EXTRACTION_MODEL` and `EXTRACTION_MODEL_LARGE` to `gemini-3.1-flash-lite-preview`. Equivalent quality, ~6x cost reduction, slightly faster.

3. **Risk**: flash-lite is a PREVIEW model. Preview models can be changed/removed without notice. Mitigations: (a) keep 2.5-flash as fallback in retry logic, (b) pin model version if possible, (c) monitor for deprecation.

4. **No need to revert to single-pass**. Split+thinking beats all prior baselines on aggregate quality and reliability.
