# FD-319 — PDP Re-Extraction Idempotency: Prompt-Level Experiment

**Date**: 2026-05-16
**Ticket**: FD-319 (Verify PDP is correctly additive across multiple discussion threads)
**Branch**: `FD-319` (btcopilot worktree, HEAD `ba817fa`). Prompt change in `fdserver/prompts/private_prompts.py` (private IP, uncommitted).
**Model(s)**: `gemini-3-flash-preview` (production extraction model), `gemini-3-pro-preview` (comparison).
**Prompts under test**: PRODUCTION (`fdserver/prompts/private_prompts.py`), loaded via `FDSERVER_PROMPTS_PATH`.

## Objective

Quantify the raw-LLM committed-duplicate rate on PDP re-extraction (deterministic
repair `fix_committed_person_duplicates` bypassed), then test a `DATA_EXTRACTION_PASS1_CONTEXT`
salience variant. Ship the prompt change only if duplicate rate drops AND F1 is not regressed.
The deterministic repair remains as a safety net regardless of outcome.

## Methodology

- **Harness**: `/tmp/fd319_dup_experiment.py`. Reconstructs the Pass-1 prompt exactly as
  `pdp._two_pass_extract` (`DATA_EXTRACTION_PASS1_PROMPT` + `DATA_EXTRACTION_PASS1_CONTEXT`,
  committed state via `pdp._committed_state_for_prompt`), calls `llmutil.gemini_structured(..., large=True)`
  directly. **No repair, no validation retry, no correction prompt** — pure first-attempt Pass-1 LLM output.
  Committed/marriage/pair-bond duplication is a Pass-1 phenomenon; Pass 2/3 (shifts/SARF) are out of scope.
- **Scenarios** (mirror `btcopilot/tests/personal/test_committed_reextract_e2e.py`):
  - `simple_people` — committed {Mary, John}+marriage; text adds new sister Sarah.
  - `simple_marriage` — committed {Mary, John}+marriage; text restates the marriage.
  - `complex` — 15-person 3-generation committed family; full re-narration.
- **Metrics** (mirror the e2e assertions): a run is a *committed-duplicate* if any committed
  person name reappears as a delta person, OR any committed marriage/death/divorce/birth
  (committed child) is re-emitted, OR any committed pair-bond dyad is re-emitted.
- **N** = 10 runs per (model × scenario), temperature 0.1, thinking budget 1024 (production config).
- **F1 gate**: `btcopilot/training/run_extract_full_f1.py`, 6 GT discussions [36,37,39,48,50,51],
  `gemini-3-flash-preview`, full 3-pass pipeline (repair + validation active), 2 runs each for
  backup (pre-variant production prompts) and variant. The 6 GT discussions are first-session
  (no committed state) so they directly test whether the variant's `NEW`-qualifier wording
  suppresses legitimate first-pass extraction (recall loss).

## Results — Raw-LLM committed-duplicate rate (N=10, no repair)

| model / scenario | Baseline (production prompt) | Variant |
|---|---|---|
| flash / simple_people | 0/10 = 0.00 | 0/10 = 0.00 |
| flash / simple_marriage | 0/10 = 0.00 | 0/10 = 0.00 |
| **flash / complex** | **10/10 = 1.00** | **0/10 = 0.00** |
| pro / simple_people | 0/10 = 0.00 | 0/10 = 0.00 |
| pro / simple_marriage | 0/10 = 0.00 | 0/10 = 0.00 |
| pro / complex | 1/10 = 0.10 | 0/10 = 0.00 |

Baseline flash/complex failure decomposition (all 10 runs identical): **0 duplicate people,
0 duplicate structural events, 0 duplicate births, exactly 2 duplicate pair_bonds**. The
residual non-idempotency on production prompts is *exclusively* committed pair-bond re-emission
by flash on a large committed state. pro/complex: 9/10 fully clean, 1 run had 3 duplicate births.

Raw data: `doc/induction-reports/2026-05-16_08-14-00--fd319-committed-dup-baseline/`,
`doc/induction-reports/2026-05-16_08-18-08--fd319-committed-dup-variant/`.

## Results — F1 gate (gemini-3-flash-preview, 6 GT discussions, avg of 2 runs)

| metric | backup | variant | Δ | backup run spread |
|---|---|---|---|---|
| people | 0.925 | 0.919 | −0.006 | 0.912 – 0.938 |
| events | 0.474 | 0.518 | +0.044 | 0.429 – 0.519 |
| pair_bonds | 0.820 | 0.845 | +0.025 | 0.802 – 0.837 |
| aggregate | 0.671 | 0.687 | +0.016 | 0.655 – 0.686 |

Every Δ is within run-to-run variance (backup's own two runs differ by 0.090 on events,
0.031 on aggregate; consistent with the documented 10–15% events stochasticity). **Defensible
claim: no F1 regression.** People recall is preserved (0.925→0.919, within noise; both variant
runs 0.917/0.921) — the `NEW`-qualifier wording does **not** suppress first-session extraction.
The apparent events/aggregate gains are NOT claimable as real signal at N=2.

## The change

Single additive edit to `DATA_EXTRACTION_PASS1_CONTEXT` in `fdserver/prompts/private_prompts.py`.
Root cause: the most-proximal generation directive (`EXTRACT: ... all pair bonds between
couples/parents ...`) had **no committed carve-out** and overrode the upstream
"COMMITTED DATA — REFERENCE, DON'T RECREATE" section + Example 6 at scale. flash obeyed the
literal "all pair bonds" instruction. Fix: qualified the directive to `All NEW ...` and added
an explicit, pair-bond-specific `⚠️ ALREADY-COMMITTED ITEMS` block with a pre-return self-check,
placed adjacent to the injected committed state. ADD-nuance only; no working content removed.

## Key findings

1. **The ticket's premise was a default-prompt artifact.** `FDSERVER_PROMPTS_PATH` is unset in
   `.env`, shell, and conftest, so the prior e2e "6/6" and the N=1 "re-emits people + marriage"
   observation ran on *default* prompts. On *production* prompts, people/marriage re-emission is
   already 0/10 on both models. (Repair is prompt-independent, so prior e2e still validates the repair.)
2. **The real residual is flash re-emitting committed pair_bonds at scale** — deterministic
   10/10 on the 15-person scenario, nothing else. pro is near-clean (1/10, unrelated).
3. **Strategy doc §2b ("Committed Data Duplication — Resolved 2026-03-05") is falsified at scale.**
   The 2026-03-05 prompt-only fix did not eliminate pair-bond re-emission on large committed states.
   Its claim that post-hoc dedup is "wrong-headed" is overturned by the FD-319 repair decision.
4. **The variant drives raw committed-dup to 0/10 across all cells and both models**, with no
   F1 regression. Decisive on the N=10 deterministic gate; F1 bounded to "not regressed" (N=2).

## Recommendation

**Ship the variant** (keep the `fdserver/prompts/private_prompts.py` edit). Decision gate met:
raw committed-dup rate 1.0→0.0 (flash/complex) and 0.1→0.0 (pro/complex), simple cells unchanged
at 0.0; F1 not regressed. **Repair stays load-bearing** — temperature-0.1 sampling on unseen
inputs gives no idempotency guarantee; the prompt reduces how often the repair must fire, it does
not make the repair redundant.

**Follow-up (non-blocking)**: fold a 3-run F1 confirmation into the next routine F1 run to
tighten the "no regression" claim; thin N=2 is the only soft spot, and the change is low-risk
(additive, PASS1_CONTEXT-only, people recall preserved).
