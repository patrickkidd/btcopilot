## Key Findings

### 1. gemini-3-flash-preview confirmed best across all providers

+6.7% Aggregate, +9.1% Events, 23% faster, only 33% more expensive ($0.004/extraction delta). No non-Google model matches its quality+speed combination.

### 2. gpt-5-mini has the highest raw Events F1 but is disqualified

Events 0.410 (+12.6% vs baseline) — highest of ANY model tested. But 460s latency (6.2x slower), 1/6 extraction failures, and weak Bonds (0.650, -20.6%) disqualify it. Worth monitoring if OpenAI improves speed.

### 3. gpt-5.2 is the only credible non-Google backup

Ties gemini-3-flash on Events (0.397), 196s latency is borderline usable. Main weakness: Bonds F1 0.514 (-37.2%) — would need prompt tuning. Best SARF scores of any non-Gemini model.

### 4. All non-Gemini models have weak Pair Bond extraction

| Model | Bonds F1 | vs baseline (0.819) |

| gpt-5.2 | 0.514 | -37.2% |

| gpt-5-mini | 0.650 | -20.6% |
| grok-4-1-fast | 0.825 | +0.7% (but N=4) |

The prompts are heavily tuned for Gemini's structured output conventions. Bond extraction is the most schema-sensitive component.

### 5. Thinking budget > model choice > provider choice

Gemini 3 Flash without thinking (Events 0.276) scores worse than every non-Gemini model with reasoning. The #1 quality lever remains thinking/reasoning capability, not model size or provider.

### 6. Non-Gemini integration requires compatibility layer

All OpenAI/Grok models needed:
- **0→None**: OpenAI strict mode forces required fields → models output `0` for optional person links
- **Positive→negative ID remapping**: All OpenAI models use positive sequential IDs
- **API param differences**: gpt-5.x/o-series need `max_completion_tokens`, no custom temperature

### 7. "Premium AI" tier is viable but cost-irrelevant

At $0.01-0.07/extraction, the API cost difference between models is invisible. A premium tier would be justified by quality (better extraction accuracy), not cost. The Gemini 3 Flash upgrade should be the default for all users — the $0.004/extraction cost increase is negligible.

## SARF Improvement Projections for Non-Gemini Models

Analysis of why non-Gemini SARF is catastrophically low and whether model-specific prompt engineering could close the gap.

### Root Cause: OpenAI Strict Mode Nullable Field Problem

OpenAI's `json_schema` strict mode requires ALL schema fields. SARF variables (symptom, anxiety, relationship, functioning) should be null when not applicable to an event, but OpenAI forces models to hallucinate values. This is likely the dominant factor — models are penalized for generating SARF values on non-shift events AND for generating wrong values when forced to guess.

Gemini's `response_schema` integrates field descriptions as generation guidance and supports nullable fields natively. OpenAI treats schema descriptions as validation metadata only — models don't benefit from the same inline extraction instructions.

### Projected Ceilings with Isolated Prompt Work

| Model | Current SARF | Projected Ceiling | Effort | Key Limitation |
|-------|-------------|-------------------|--------|----------------|
| gpt-5.2 | 0.102 | 0.20-0.30 | Medium | Best structured output; nullable workaround (add "not_applicable" enum values) + prompt restructure could 2-3x SARF. Still ~50% below Gemini baseline. |
| gpt-5-mini | 0.092 | 0.18-0.25 | Medium | Similar ceiling to gpt-5.2 but weaker on complex schemas, 460s latency unchanged. |
| o4-mini | 0.080 | 0.15-0.20 | High | Chain-of-thought reasoning doesn't help with schema-level problems. Latency already 289s. |
| grok-4-1-fast | 0.082 | 0.10-0.15 | High | Weakest structured output support, 2/6 extraction failures. Limited improvement potential. |

### Recommendation

The low-hanging fruit is a nullable enum workaround (adding "not_applicable" values to SARF enums in the OpenAI schema). This is a one-time change to the compatibility layer, not per-model prompt work. Even with this fix, non-Gemini models would likely plateau at 0.20-0.30 SARF avg — still 40-60% below Gemini baseline. Deep per-model prompt engineering is not worth pursuing given the structural disadvantage.

## Recommendations

1. **Upgrade to gemini-3-flash-preview with thinking=1024** for all users. The quality improvement (+6.7% Aggregate, +9.1% Events) at $0.004/extraction more cost is a no-brainer.

2. **Do not implement a "premium AI" tier based on model switching.** The cost differences are negligible ($0.01-0.07/extraction). Quality differences are prompt-dependent, not model-dependent. All users should get the best available model.

3. **Keep gpt-5.2 as documented backup** in case Gemini becomes unavailable. Would require prompt adaptation for bonds and SARF.

4. **Monitor gpt-5-mini** — if OpenAI improves latency, its Events F1 (0.410) makes it the strongest alternative.

5. **Multi-run validation required** before production deployment of gemini-3-flash-preview. All results here are single-run with 10-15% variance.
