# Workstream Engine — Requirements
*2026-05-23. Implementation-agnostic. Describes what the system must do, not how.*

Implementation plan: `btcopilot/doc/plans/2026-05-23--workstream-engine/PLAN.md`

---

## 1. Parallel Isolation

- Multiple workstreams can run concurrently without interference — in code, in filesystem, in running processes, and in test data.
- Each workstream's file changes, running servers, fixture data, and test state are fully independent from every other workstream.
- The system can tell Patrick, provably (not by assertion), that a workstream's sandbox is isolated and origin clones are untouched.

---

## 2. Single Entry Point and Resumability

- One command handles all workstream stages: planning, implementing, testing, reviewing.
- Re-invoking the command from any CC session at any time resumes the workstream at exactly the correct state.
- No prior conversation history is required to resume — all state is stored durably and reconstructed on entry.
- Drift within a session is recoverable by re-invoking, not catastrophic.

---

## 3. Deterministic Stage Enforcement

- Stage transitions, output validation, and sandbox management are executed by code — not dependent on whether the LLM follows an instruction.
- Rules with machine-verifiable properties are enforced automatically against sub-agent output, not by asking the LLM to check them.
- Enforcement triggers automatically when sub-agent output is produced — not by Claude deciding to call a validator.
- If validation fails, the sub-agent is re-prompted with the specific failure before results are presented to Patrick.

---

## 4. Self-Improving Rules That Scale

- Corrections from Patrick are stored as addressable, structured, prunable rules — never appended to skill files.
- Each rule belongs to a specific stage.
- A new rule takes effect immediately in the same session without restarting anything.
- Rules that supersede old ones remove the old ones — no unbounded accumulation.
- The skill file stays small and contains only routing logic, regardless of how many rules accumulate.
- Adding or removing a rule with a machine-verifiable property automatically adds or removes its enforcement.

---

## 5. Autonomous Implementation

- Implementation runs with minimal human interaction as the explicit goal.
- The system surfaces blockers to autonomous running explicitly, with a measurement of how well it can run long-horizon.
- Wall-clock ETA is displayed and improves over successive workstreams via recorded telemetry.
- TODO burn-down status is visible.
- A PR is created for each repo in scope. A mandatory Jira comment is posted on the ticket immediately after PR creation.
- Agent-predicted failure modes are generated by scanning the code diff before handoff to Patrick.

---

## 6. Sandboxed Testing

- The test environment is provably isolated: worktree paths read from durable state (not derived), origin clones verified clean, sandboxed server on an isolated port, ephemeral database with no shared state with production.
- Production server and database are never touched during any workstream stage.
- `close_all_instances()` is called before every app launch — no leftover processes.
- Fixture data generation is scenario-driven and methodology-based, not ad-hoc per test. Scenarios cover all PDP card types, edge cases, and conflict-inducing data.
- Fixture names are realistic and synthetic — never "Test User" or placeholder strings.

---

## 7. Structured Output Validation

- Sub-agent output has a defined schema per stage. Required fields are checked before results are accepted.
- The human validation script is validated structurally: correct number of steps, required keys on each step.
- `sandbox_confirmed` must be present and true before testing results are accepted.
- PR URLs and Jira comment confirmation must be present before implementation advances.

---

## 8. Human Validation Script

- A human validation script is always produced and presented inline — never in a file, never omitted, even if all automated tests pass.
- Format: numbered steps, each with Arrange / Act / Assert in plain language.
- Length: 1–7 steps.
- Every Assert names a specific observable UI state change — not inferred, not "should be saved."
- At least one step verifies data persistence by opening a different view or app.
- The script is a living document: passed items are removed, new issues add items.

---

## 9. Bug Tracking and Remediation

- Human-discovered failure modes are logged per workstream with: description, severity, repro steps, screenshot path.
- Each failure mode has a validation status: agentic-verified, human-verified, or unverified. Human verification is the final gate.
- Bugs are filed as child Jira tickets, gated on explicit approval per session.
- During the grind session, the agent burns down bug fixes in the worktree simultaneously while Patrick walks the human script — not waiting.

---

## 10. Visibility

- A kanban of all active workstreams is available on demand: ticket, summary, stage, blockers, tasks remaining, PR count.
- A detail view per workstream shows: stage, worktree path, repos, blast radius, blockers, open questions, tasks remaining, PR URLs, failure mode list with validation status, decisions.
- The detail view is consistent in schema across all workstreams.

---

## 11. Telemetry and ETA

- Every stage transition records a timestamp.
- Stage history is retained per workstream.
- After sufficient historical data, ETA is computed from past durations and improves over time.
- ETA is segmented by variables that affect duration: repos in scope, number of ACs, LOC changed, number of blockers.

---

## 12. Safety

- Origin clones are always on master and never modified during any workstream stage. This is verified, not asserted.
- All feature work lives in worktrees at a canonical path enforced by the system.
- The skill owns worktrees. Worktrees are removed automatically when the workstream advances to `done` — Patrick is never asked to tear them down.
- Before advancing to `done`, PRs must exist or Patrick must explicitly say "discard."
- Sandboxed servers run on isolated ports — never the production port.
- Sandbox is stopped automatically when advancing to `done`.

---

## 13. Planning Stage Output

- Acceptance criteria are surfaced and displayed before any work begins.
- Open questions are tracked explicitly.
- A tentative plan is produced and stored with a timestamp.
- Risks, blockers, and blast radius of the tentative plan are assessed and recorded.
- Clarifying questions are asked in a single message and answered before any code is written.
- The Jira ticket is the output artifact of this stage — it must exist and be in scope before implementation begins.

---

## 14. Review Stage Output

- Review is conducted against Jira acceptance criteria — not implementation aesthetics.
- Verdict is APPROVED or NEEDS_WORK, with blocking vs. non-blocking issue lists.
- DiagramData field sync, Jira comment presence, and test log existence are all verified.

---

## 15. Project Agnosticism

- The engine is project-agnostic. Project-specific knowledge lives in swappable rules and documentation.
- The same engine can be used across multiple projects by swapping the rules set and project documentation.
- Micron-scale multi-user deployment is a future concern and must not drive current implementation decisions.

---

## 16. AC Maintenance

- When the skill file is improved or any requirement evolves, this document is updated in the same session — never deferred.
- The skill file and this document are kept in sync: every behavioral constraint the skill enforces has a corresponding AC here.
